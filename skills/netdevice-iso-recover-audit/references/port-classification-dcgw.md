# DCGW 端口分类规则（已实现）

本规则针对**核心层 DCGW**（设备名包含 `S-RT` 或 `E-RT`）。规则依据 LLDP 邻居的对端设备名称，对本端物理端口做上/下/互联/剩余四类划分。

## 分类规则

| 端口类别 | 判定依据（对端设备名包含以下任一关键字） |
| -------- | ---------------------------------------- |
| **上行端口** | `CBN`（广电防火墙）、`PC-CMNET` 或 `N-RT`（CMNET 防火墙 / CMNET CE）、`NETNMS`（网管 CE）、`NETCN`（业务 CE）、`NETBOSS`（业支 CE）、`S-FW`（业务业支防火墙）、`M-FW`（网管防火墙）、`M-EOR`（管理 EOR） |
| **下行端口** | `S-EOR`（业务 EOR）、`LB`（负载均衡） |
| **互联端口** | `S-RT` 或 `E-RT`（对端也是 DCGW） |
| **剩余端口** | 上述三类规则都未命中 |

> 「业务业支防火墙」与「CMNET 防火墙」关键字均为 `S-FW`，因此上行端口的判定应仅按对端名称是否包含 `S-FW`；具体的精细区分不影响上行分类。

## 重要约束

1. **聚合口属性继承**：聚合口（`Eth-Trunk` / `Port-Channel` / `Bridge-Aggregation` 等）的上/下/互联属性与其任意一个成员物理口保持一致；本端未单独出现在 LLDP 中的物理口若属于该聚合口，则跟随聚合口属性。
2. **对端名称包含多关键字**：以「最先命中」为准，匹配顺序为上行 → 下行 → 互联 → 剩余。
3. **本端出现但未在 LLDP 中**：视为「剩余端口」，需运维人员人工复核。
4. **未出现对端信息**（孤立端口）：同样归入「剩余端口」。

## 防火墙主备识别（v6）

DCGW / CMNET CE 等双防火墙接入场景下，需要在隔离 / 恢复时进一步区分「主防火墙」与「备防火墙」。v6 在 v5 基础上引入**防火墙对（fw_pair_id）**概念，并针对 S-FW 新增**数字接近度配对**算法。

核心设计：

1. **防火墙对**：每对防火墙由一台主（-01）和一台备（-02）组成，共享同一个 `fw_pair_id`。
2. **主备角色**：`fw_role` 为 `主`（-01 结尾）或 `备`（-02 结尾）。
3. **子顺序检查范围**：每对防火墙内部，检查本端所有连接该对端防火墙的端口是否满足「先全部备边端口，再全部主边端口」（隔离）或「先全部主边端口，再全部备边端口」（恢复）。
4. **跨对顺序自由**：不同防火墙对之间无顺序约束（例如 S-FW 对和 CBN 对可以交错出现）。

### 防火墙对配对算法（v6 新增）

#### 步骤一：按类型分流

根据对端设备名包含的关键字，将防火墙分为两类：

| 类型 | 判断条件 | 配对方式 |
| ---- | -------- | -------- |
| **M-FW / CBN / C-FW** | 对端设备名含 `M-FW` 或 `CBN` 或 `C-FW` | 按 `-01`（主）/ `-02`（备）后缀直接配对 |
| **S-FW** | 对端设备名含 `S-FW` 但不含 `M-FW` / `CBN` / `C-FW` | 按数字接近度配对（见下方算法） |

> `S-FW` 同时对应「业务业支防火墙」和「CMNET 防火墙」，没有直接标志区分。v6 引入的数字接近度算法专门解决这类设备的配对问题。

#### 步骤二：S-FW 数字接近度配对算法

当 LLDP 中出现多台 S-FW（例如 4 台、6 台），按以下步骤两两配对：

1. **收集所有 S-FW 设备**（先去重，同一台防火墙可能在多个成员口 LLDP 中重复出现），提取其设备名中的数字标识（格式：**任意 2 位大写字母** + `<数字>` + `-S-FW-<01|02>`，正则 `r"[A-Z]{2}(\d+)-S-FW-(\d+)"`）。
   - v6.1 起放宽前缀匹配：原 v6 仅识别 `(AE|AF)`，但实际 DCGW 拓扑中备机 S-FW 经常使用 `AD` / `AC` 等其它两位大写字母前缀。只识别 AE/AF 会导致这些设备 `s_fw_num=None`、无法进入配对，v6 子顺序检查全部失效。
   - 例如 `NFV-D-HNZJI-02A-1801-AE09-S-FW-01` → 数字 = `09`，角色 = 主（-01）
   - `NFV-D-HNZJI-02A-1801-AF08-S-FW-02` → 数字 = `08`，角色 = 备（-02）
   - `NFV-D-HNGZ-14A-3203-AD14-S-FW-02` → 数字 = `14`，角色 = 备（-02）（v6.1 起也识别）
   - `NFV-D-HNZJI-01A-1805-AC03-S-FW-01` → 数字 = `03`，角色 = 主（-01）（v6.1 起也识别）
2. **按数字将 S-FW 分组**：数字相同的分为一组（如 04 组、09/08 组）。
3. **组内配对**：每组内，`-01` 与 `-02` 配成一对。若某组内只有 `-01` 或只有 `-02`，视为**未配对**，不参与主备子顺序检查。
4. **跨组配对**（仅当某组内只有一台设备时触发）：从数字最接近的组中寻找配对，例如：
   - 假设数字分布为：04 组（-01 有，-02 有）、09 组（-01 有）、08 组（-02 有）
   - 09 组（-01）尝试与 08 组（-02）配对（因为 09 与 08 差值最小）
   - 04 组（-01/-02）自行配对

**配对示例**（来自实际设备 NFV-D-HNZJI-02A-1801-AE02-S-RT-01 的 LLDP）：

| 对端设备名 | 数字 | 角色 | 配对结果 |
| --- | --- | --- | --- |
| `NFV-D-HNZJI-02A-1801-AE09-S-FW-01` | 09 | 主 | **Pair-S-FW-1**（与 AF08-S-FW-02 配对，差值 1） |
| `NFV-D-HNZJI-02A-1801-AF08-S-FW-02` | 08 | 备 | **Pair-S-FW-1** |
| `NFV-D-HNZJI-02A-1801-AE04-S-FW-01` | 04 | 主 | **Pair-S-FW-2**（与 AF04-S-FW-02 配对，差值 0） |
| `NFV-D-HNZJI-02A-1801-AF04-S-FW-02` | 04 | 备 | **Pair-S-FW-2** |

> **为什么用数字接近度而非精确相等**：业务业支防火墙（AE/AF-04）和 CMNET 防火墙（AE/AF-09/AE/AF-08）在网络上处于不同平面，但设备型号命名可能接近。用数字接近度配对可以在无明确业务标签时作为保守的启发式配对方法。

#### 步骤三：M-FW / CBN / C-FW 配对

对于 `M-FW` / `CBN` / `C-FW` 类型，按 `-01`（主）/ `-02`（备）后缀直接配对，不需要数字接近度算法。

### 触发条件

当且仅当以下两个条件同时满足时，本端接口才进入「主备识别」流程：

1. **对端设备属于防火墙类**——对端设备名包含以下任一关键字：
   - `S-FW`（业务业支防火墙 / CMNET 防火墙；与 `references/device-types.md` 对齐）
   - `M-FW`（网管防火墙）
   - `C-FW`（CMNET 防火墙）
   - `CBN`（广电防火墙，例：`YDCBN-FW`）

   > **不**列入防火墙关键字：`PC-CMNET`、`N-RT`——这两者是 CMNET CE（出口路由器）的关键字，**不是**防火墙。误列会让 DCGW 把去往 CMNET 路由器的上行口也当作主备候选。

2. **配对成功**——该防火墙设备按上述配对算法找到了配对的另一台（主 + 备同时存在）。仅出现其一不视为配对。

### 主备识别规则

| 对端防火墙设备名后缀 | fw_role |
| --- | --- |
| `-01` | **主** |
| `-02` | **备** |
| 其他（如 `-03`、`-A`、`-B`、无后缀） | `None`（不参与主备子顺序） |

> **为何选 `-01`/`-02`**：这是网络云资源池现行主流命名约定（`AE04-S-FW-01` / `AF04-S-FW-02`、`AE15-YDCBN-FW-01` / `AF14-YDCBN-FW-02` 等），在 AE / AF 双平面双归属组网中明确标识「主用 / 备用」角色。后续如厂家引入新后缀（如 `Master` / `Slave`），需要同步在此表追加一行。

### v6 数据结构扩展

v6 中 `PortInfo` 新增 `fw_pair_id` 字段，用于标识本端端口所属的防火墙对：

| 字段 | 取值 | 说明 |
| --- | --- | --- |
| `category` | 上行 / 下行 / 互联 / 剩余 | 决定进入 expected_order 的哪一段 |
| `fw_role` | 主 / 备 / None | 主备子顺序的检查依据 |
| `fw_pair_id` | `Pair-S-FW-1` / `Pair-S-FW-2` / `Pair-CBN-1` / … / None | 本端端口所属的防火墙对标识；同对内所有本端端口共享同一个 `fw_pair_id` |

> `fw_pair_id` 用于稽核报告中标注「所属防火墙对」，方便审计人员核查每对本端端口的操作是否连续、是否符合子顺序要求。

### 主备子顺序（v6：按对检查，隔离「先全部备边端口，再全部主边端口」/ 恢复「先全部主边端口，再全部备边端口」）

v6 的核心改变：**子顺序的检查粒度从「类型分组」细化为「每对防火墙」**。

#### 核心原则

| 操作 | 子顺序要求 | 说明 |
| --- | --- | --- |
| **隔离** | 先关闭所有连接**备防火墙**的本端端口，再关闭所有连接**主防火墙**的本端端口 | 业务短暂走单「主」，再断「主」，避免主备同时断开导致业务真空 |
| **恢复** | 先打开所有连接**主防火墙**的本端端口，再打开所有连接**备防火墙**的本端端口 | 业务先恢复主路径，再开备补回热备，避免备先开时业务单边承担后又双主不一致 |

#### 检查逻辑（v6）

对每个防火墙对（fw_pair_id），在脚本的上行段和下行段内：

1. **隔离**：该对内所有 `备` 角色的接口操作必须出现在所有 `主` 接口操作**之前**。一旦出现第一个 `主`，该对后续不能再出现 `备`（即所有 `备` 连续，所有 `主` 连续，且 `备` 段在前）。
2. **恢复**：该对内所有 `主` 角色的接口操作必须出现在所有 `备` 接口操作**之前**。一旦出现第一个 `备`，该对后续不能再出现 `主`。

#### 发现项必须同时给出建议正常顺序（v6.2）

检测到某对的主备子顺序异常时，**发现项中必须额外列出该对在本脚本中应有的「建议正常操作顺序」**（用本脚本中实际出现的接口 + 行号，按行号升序列出），格式示例：

> 防火墙对 Pair-S-FW-1 主备子顺序异常：隔离应「先全部备边端口再全部主边端口」，但 行 32(Eth-trunk 21) 出现在主防火墙操作之后。建议的正常操作顺序：先全部备边端口（行 12(Eth-Trunk6)、行 15(Eth-Trunk9)），再全部主边端口（行 8(Eth-Trunk5)、行 22(Eth-Trunk20)）。

要求：
- 隔离场景：「先全部备边端口（行 X(...) 、 行 Y(...)），再全部主边端口（行 Z(...) 、 行 W(...)）」
- 恢复场景：「先全部主边端口（行 X(...) 、 行 Y(...)），再全部备边端口（行 Z(...) 、 行 W(...)）」
- 跨对顺序自由（v6 放宽约束），此处只约束**单对内部**的总顺序；不需要限定两对之间的先后。

理由：抽象的「应先备后主」不足以让审计人员直接重排脚本；列出该对在**本脚本中**应有的具体行号顺序后，审计人员可以据此直接修改。

#### 与 v5 的关键差异

| 维度 | v5 | v6 |
| --- | --- | --- |
| 分组粒度 | 按防火墙**类型**（S-FW / M-FW / C-FW / CBN）全局检查 | 按防火墙**对**（Pair-S-FW-1 / Pair-S-FW-2 / Pair-CBN-1 …）逐对检查 |
| 子顺序检查范围 | 同类型内 first_master / first_backup 的首位置比较 | **该对内所有备边端口** vs **该对内所有主边端口**的连续性检查 |
| 跨对顺序 | 同类型内严格（S-FW 所有主不能在所有备之前） | **跨对顺序自由**（Pair-1 的主可以和 Pair-2 的备交错） |
| S-FW 配对 | 全局 -01=主 / -02=备，无配对概念 | **按数字接近度两两配对**，每对独立检查 |

> v6 放松了跨对顺序约束，更符合实际业务场景：DCGW 可能同时连接多对防火墙，每对的隔离/恢复操作相对独立，只有**同对内的主备顺序**需要严格遵守。

#### 典型正确示例（v6）

假设 Pair-S-FW-1（AE04-S-FW-01/AF04-S-FW-02）和 Pair-CBN-1（AE15-YDCBN-FW-01/AF14-YDCBN-FW-02）：

```
隔离脚本（上行段）：
  Eth-Trunk5   → S-FW-01 主（Pair-S-FW-1）
  Eth-Trunk6   → S-FW-02 备（Pair-S-FW-1）
  Eth-Trunk8   → CBN 主（Pair-CBN-1）         ← 允许跨对交错
  Eth-Trunk9   → CBN 备（Pair-CBN-1）
  Eth-Trunk21  → S-FW-02 备（Pair-S-FW-1）   ← 同对内，所有备已连续（在主之前）
  Eth-Trunk20  → S-FW-01 主（Pair-S-FW-1）   ← 同对内，所有主在所有备之后
```

#### 典型错误示例（v6）

```
隔离脚本（上行段）：
  Eth-Trunk8   → CBN 主（Pair-CBN-1）
  Eth-Trunk9   → CBN 备（Pair-CBN-1）        ← 主在备之前，违反「先备后主」
```

```
隔离脚本（上行段）：
  Eth-Trunk5   → S-FW-01 主（Pair-S-FW-1）   ← 主先出现
  Eth-Trunk6   → S-FW-02 备（Pair-S-FW-1）   ← 主之后出现备，违反「所有备在主之前」
```

### 未配对防火墙的处置

- 如果 LLDP 中只有 `-01`、没有 `-02`（或反之），**不视为配对**：
  - 该防火墙端口 `fw_role` 为 `None`，不参与主备子顺序检查。
  - `audit.py` 会在发现项中追加一条提示：`「检测到防火墙 <name>，但未在 LLDP 中找到配对的 -01/-02，需人工确认主备关系」`，提醒人工复核。
- 防火墙主备子顺序**仅在配对成功时生效**；未配对时按普通上 / 下行处理。

## 变更记录

- **v6.2（当前）**：
  1. **发现项必须同时给出建议正常顺序**：当检测到某防火墙对的主备子顺序异常时，发现项中除了「哪些接口出现在错误的角色后面」之外，还必须额外列出该对在本脚本中应有的「建议正常操作顺序」（用本脚本中实际出现的接口 + 行号）。目的是让审计人员可以据此直接调整脚本，不停留在抽象描述。
  2. 实现位置：`scripts/audit.py` 的 `audit_order()` 中，对每对 `pair_id` 处理 `backup_after_master` / `master_after_backup` 时构造「先全部备、再全部主」或「先全部主、再全部备」的具体行号列表。
- **v6.1（前一版）**：
  1. **S-FW 前缀匹配放宽**：`_S_FW_NUM_RE` 由 `(AE|AF)(\d+)-S-FW-(\d+)` 改为 `[A-Z]{2}(\d+)-S-FW-(\d+)`，使 `AD` / `AC` 等任意 2 位大写字母前缀的 S-FW 也能进入配对。修复「备机 S-FW 用 AD 前缀但 v6 子顺序检查完全跳过」的实际场景。
  2. **配对前去重**：`detect_fw_pairs()` 入口对防火墙邻居按设备名去重（同一台防火墙可能在多个成员口 LLDP 中重复出现），修复放宽正则后暴露的「同一对生成多对 Pair」重复 bug。`M-FW` / `CBN` / `C-FW` 路径同步受益。
- **v6**：
  1. **防火墙对（fw_pair_id）**：新增 `fw_pair_id` 字段标识本端端口所属的防火墙对，实现「每对防火墙独立检查」的设计目标。
  2. **S-FW 数字接近度配对算法**：当 LLDP 中出现多台 S-FW 时，按设备名中的数字接近度两两配对（而非 v5 的全局 -01=主 / -02=备）。例如 AE09-S-FW-01 与 AF08-S-FW-02（差值1）配对，AE04-S-FW-01 与 AF04-S-FW-02（差值0）配对。
  3. **子顺序检查强化**：从 v5 的「first_master < first_backup」首尾检查升级为「该对内所有备在所有主之前（隔离）/ 所有主在所有备之前（恢复）」的连续性检查，避免主备交替出现的中间状态被漏检。
  4. **跨对顺序放松**：v5 要求同类型内所有主不能在所有备之前，v6 改为跨对顺序自由（同对内主备严格按顺序，不同对之间可以交错），更符合 DCGW 同时连接多对防火墙的实际场景。
- **v5（已废弃）**：
  1. 主备子顺序的**作用域**从「上行」扩到「上行」**与**「下行」两段（v4 错误地只作用于上行；现实是 CMNET CE 这类设备走下行连防火墙）。
  2. 主备子顺序的**分组粒度**从「全局」改为「按防火墙类型分组」（v4 错误地用全局主备子顺序把不同类防火墙混在一起要求子顺序；v5 按 `C-FW` / `CBN` / `M-FW` / `S-FW` 分组，组内独立检查，跨组顺序自由）。
  3. 隔离与恢复采用**相反**的子顺序：隔离「先备后主」、恢复「先主后备」。v4 早期误把两者都定成「先主后备」，本版本回归用户最初规则。
- **v4（已废弃）**：
  1. 主备子顺序的**作用域**从「上行」扩到「上行」**与**「下行」两段（v4 错误地只作用于上行；现实是 CMNET CE 这类设备走下行连防火墙）。
  2. 主备子顺序的**分组粒度**从「全局」改为「按防火墙类型分组」（v4 错误地用全局主备子顺序把不同类防火墙混在一起要求子顺序；v5 按 `C-FW` / `CBN` / `M-FW` / `S-FW` 分组，组内独立检查，跨组顺序自由）。
  3. 隔离与恢复采用**相反**的子顺序：隔离「先备后主」、恢复「先主后备」。v4 早期误把两者都定成「先主后备」，本版本回归用户最初规则。
- **v4**：将 v3 的「上行-主防火墙 / 上行-备防火墙」从 `category` 的子分类**下沉**为 `PortInfo.fw_role` 附加字段；主备子顺序仅对「上行」段内 fw_role 作检查；修正 v3 中把 `PC-CMNET` / `N-RT` 误列为防火墙关键字的错误。但 v4 的「全局主备子顺序」语义与业务实际不符（不同类防火墙应独立检查；隔离/恢复也应是相反的子顺序），已被 v5 废弃。
- **v3**：曾把「上行-主防火墙 / 上行-备防火墙」作为 `subcategory` 字段，**但**这破坏了跨大类的顺序骨架（让子分类与「下行/互联」并列），与设计本意不符，已被 v4 废弃。
- **v2**：`M-EOR`（管理 EOR）从「下行」调整为「上行」。理由：管理 EOR 实际上是 DCGW 上连到网管网/管理平面的网关，整机隔离时与外联 CE / 防火墙同样需要先断，否则可能出现『业务已隔离但仍能纳管』的半隔离状态。
- **v1（历史）**：`M-EOR` 归在「下行」段。
