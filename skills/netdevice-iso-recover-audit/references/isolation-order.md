# 网络设备隔离/恢复顺序定义

> 隔离（shutdown）与恢复（undo shutdown）必须按本文件定义的安全顺序执行，避免对现网业务造成影响。

## 通用原则

| 操作 | 推荐顺序 |
| ---- | -------- |
| **隔离（shutdown）** | 先断「上行」（离开现网）→ 再断「下行」（脱离业务）→ 最后断「互联」（脱离对端设备） |
| **恢复（undo shutdown）** | 先开「互联」（先和对端打通）→ 再开「上行」（重新接入外网）→ 最后开「下行」（恢复业务流量） |

> 顺序背后的逻辑：隔离时由外向内收敛，恢复时由内向外开放，保证任意瞬间不会出现「孤立半边」的不一致状态。

## DCGW（已实现）

| 操作 | 顺序 |
| ---- | ---- |
| 隔离 | **上行 → 下行 → 互联** |
| 恢复 | 互联 → **上行 → 下行** |

DCGW 的端口分类规则详见 `port-classification-dcgw.md`。

### 防火墙主备子顺序（v6 / v6.1，DCGW 已实现）

v6 在 v5 基础上将子顺序检查粒度从「类型分组」细化到「每对防火墙」：

1. **适用段**：上行 + 下行（与 v5 一致）。
2. **分组粒度**：从「按类型关键字分组」改为「**按防火墙对（fw_pair_id）**」逐对独立检查。不同防火墙对之间顺序自由。
3. **S-FW 配对**：通过数字接近度算法两两配对（详见 `port-classification-dcgw.md`），每对独立检查子顺序。
4. **子顺序检查强化**：从 v5 的「first_master < first_backup」首尾检查升级为「该对内所有备在所有主之前（隔离）/ 所有主在所有备之前（恢复）」的连续性检查。
5. **v6.1 修订**：S-FW 前缀匹配由 `(AE|AF)` 放宽到 `[A-Z]{2}`，配对入口按设备名去重（修复「同一台防火墙被多个成员口 LLDP 重复列出、生成多对 Pair」的 bug）。详见 `port-classification-dcgw.md` 变更记录。

| 适用段 | 操作 | 对内子顺序 |
| --- | --- | --- |
| 上行 + 下行 | **隔离** | 先关闭所有连接**备防火墙**的本端端口，再关闭所有连接**主防火墙**的本端端口 |
| 上行 + 下行 | **恢复** | 先打开所有连接**主防火墙**的本端端口，再打开所有连接**备防火墙**的本端端口 |

主备识别规则、S-FW 数字接近度配对算法与「配对失败退化」的说明见 `port-classification-dcgw.md` 的「防火墙主备识别（v6）」章节。

#### 设计理由

- **隔离与恢复采用相反的子顺序**：
  - 隔离「**先全部备边端口，再全部主边端口**」：先断「备」使业务短暂走单「主」、再断「主」，避免「主备同时断开」导致业务真空。
  - 恢复「**先全部主边端口，再全部备边端口**」：先开「主」使业务先恢复主路径、再开「备」补回热备，避免「备先开」时业务单边承担后又双主短暂不一致。
- **按对独立检查**：不同防火墙对之间的顺序无约束（例：Pair-S-FW-1 的主可以出现在 Pair-CBN-1 的备之前），但**同一对防火墙内部**必须严格遵守「所有备在前/所有主在后」（隔离）或「所有主在前/所有备在后」（恢复）的连续性要求。

#### 未配对 / 无防火墙 / 互联 / 剩余段内的防火墙端口的处置

- **未配对防火墙**（LLDP 中只有 `-01` 或只有 `-02`，或 S-FW 数字接近度无法配对）：该防火墙端口 `fw_role` 为 `None`，不参与子顺序检查；`audit.py` 会在发现项中提示人工确认。
- **无防火墙**：上行段退化为「上行-其他 → 下行 → 互联」（隔离）或「互联 → 上行-其他 → 下行」（恢复），与 v2 版本语义一致。
- **互联 / 剩余段内出现防火墙端口**（理论场景）：**不应用**子顺序，仍按该段的大类顺序处理。
